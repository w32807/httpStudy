1. 클라이언트 <-- 인터넷 --> 서버
    1) 클라이언트와 서버는 인터넷으로 연결되어 통신한다.

2. IP (Internet Protocol) - OS 7계층에서 3계층인 네트워크 계층에 해당하는 프로토콜
    [내용]
        1) 각 컴퓨터마다 IP Address를 부여받는다.
        2) 패킷(데이터그램)이라는 단위로 데이터를 전달한다.
        3) 마치 택배를 싸듯이 데이터를 출발, 목적 IP로 감싸서 데이터를 전송한다.
    [IP 프로토콜의 한계]
        1) 비연결성 - 패킷을 받을 대상이 없거나 서비스 불능상태여도(PC가 꺼져있어도) 패킷 전송
        2) 비신뢰성 - 중간에 패킷이 사라지거나, 패킷이 순서대로 안와도 (데이터가 뒤죽박죽) 상관없이 그냥 보냄
        3) IP만 있다면 같은 서버(같은 PC)에서 통신하는 App이 2개 이상이면 프로그램 구분을 할 수 없음

3. TCP (Transmission Control Protocol) - 전송제어 프로토콜
    [내용]
        1) IP 프로토콜의 단점을 보완한다.
        2) IP 프로토콜에 TCP 패킷이 감싸진다.
        3) 프로토콜 계층 안에서 클라이언트 -> 서버 데이터 전달 순서 (만들 때는 상위 계층에서 하위계층으로 가면서 데이터가 프로토콜 정보들로 감싸진다.)
            a) 애플리케이션 계층에서 데이터 생성
            b) Socket 라이브러리를 통해 TCP 계층으로 전달
            c) TCP 계층에서 TCP 정보 생성 및 메시지 데이터 포함
            d) IP 계층으로 전달 후, IP 패킷 생성 및 TCP 데이터 포함
            e) 네트워크 인터페이스 계층 (랜카드 등)에서 EtherNet 프레임으로 감싼 후 인터넷을 통해 서버쪽으로 전달
            f) 서버 쪽에서는 각 프로토콜 정보로 감싸진 패킷을 하나씩 벗겨가며 수신
        4) IP 패킷 정보 : 출발지 IP, 목적지 IP, 기타..
        5) TCP 세크먼트 정보 : 출발지 PORT, 목적지 PORT, 전송제어, 순서, 검증정보..
            >> IP의 한계를 해결하기 위한 정보가 추가됨
    [특징]
        1) 연결지향 - TCP 3 way handshake (선 연결 후 데이터 보냄)
            ** 3 way handshake (SYN : 접속요청, ACK : 요청수락)
                a) 클라이언트에서 SYN를 서버로 보냄 (서버야 살아있니?)
                b) 서버에서는 SYN + ACK를 보냄 (살아있음. 너도 살아있니?)
                c) 다시 클라이언트에서 ACK를 보냄 (응. 살아있어)
                d) 비로소 데이터 전송이 일어남
                e) 실제 물리적인 연결이 아닌, 가상(논리)적으로 연결이 된 것임
                    >> 중간의 노드들은 실제 연결이 되었는지는 모름
        2) 데이터 전달 보증 / 패킷 순서 보장함
            ** 패킷의 순서 보증
                a) 데이터를 받는 서버쪽에서 패킷의 순서가 잘못됐다고 판단하면
                    클라이언트에 다시 순서대로 보내라고 요청함

4. UDP (User Datagram Protocol)
    [내용]
        1) 기능이 거의 없음
        2) TCP와 비교하여 데이터 전달 및 순서가 보장되지 않지만 단순하고 빠름
        3) TCP는 최적화하기 어렵지만, App 수준에서 UDP 프로콜을 최적화가 용이하다.
    [특징]
        1) 연결지향 - TCP 3 way handshake 없음
        2) 데이터 전달 보증 없음 / 순서보장 없음
        3) IP와 거의 같지만 Port 정보 추가된 정도

5. Port
    [내용]
